webpack流程及原理

发表于 2020-06-15 更新于 2020-06-28 分类于 工具

从概念上讲：webpack是一个现代JavaScript应用的静态模块打包工具。 提到webpack，就不得不提及前端两个重要的概念：模块和打包。

<!-- more -->

模块：

  • 现在的前端开发都是基于模块化开发的，比如一个项目中除了最重要的.js文件之外，还有各种.css, .less, .png文件以及依赖的外部库。这种模块化方式大大方便了开发者，但是对于浏览器来说只认识.html文件，并不能处理其他各种文件。那么就需要一个工具将他们组合起来，并且能处理好他们之间的复杂关系。这就是webpack的作用。
  • 关于模块化的一下标准，如：AMD, CMD, CommonJS, ES6. （留个坑，后面深入学习模块化）

打包：

  • 有了模块化的概念，打包就容易理解了，就是将webpack中的各种资源模块进行打包合并成一个或多个包(Bundle)。
  • 并且在打包的过程中，还可以对资源进行处理，比如压缩图片，将scss转成css，将ES6语法转成ES5语法，将TypeScript转成JavaScript等等操作。

webpack运行机制

webpack的运行过程可以简单概述为如下流程： 初始化配置参数 -> 绑定事件钩子回调 -> 确定Entry逐一遍历 -> 使用loader编译文件 -> 输出文件

webpack事件流

webpack就像工厂里的一条生产线，它是一系列流程化的结果，每个流程的职责是单一的，且每个流程相互依赖。插件就是在特定的流程对资源做处理，所以插件只监听他关心的事件就可以了。webpack通过<font color="red">Tapable</font>组织这条流水线. Tapable的原理通过发布者-订阅者模式实现,核心代码类似：

前端安全：XSS与CSRF

发表于 2020-06-02 分类于 安全

前端安全是一个很值得关注的焦点，现对XSS与CSRF稍做总结.

<!-- more -->

XSS

XSS简介

一言以蔽之：XSS攻击就是页面被注入了恶意的代码 。利用这些恶意代码，攻击者可以获取用户的Cookie等，危害数据安全。至于通过哪些方法"注入"？来看XSS分类：

XSS分类

根据攻击来源，XSS攻击可分为反射型，存储型，DOM型三种。

存储型

存储型XSS的攻击步骤：

  1. 攻击者将恶意代码提交到目标网站的数据库中。
  2. 用户打开目标网站时，网站服务端将恶意代码从数据库取出，拼接在 HTML 中返回给浏览器。
  3. 用户浏览器接收到响应后解析执行，混在其中的恶意代码也被执行。
  4. 恶意代码窃取用户数据并发送到攻击者的网站，或者冒充用户的行为，调用目标网站接口执行攻击者指定的操作。 这种攻击常见于带有用户保存数据的网站功能，如论坛发帖、商品评论、用户私信等。

反射型

反射型XSS的攻击步骤：

  1. 攻击者构造出特殊的 URL，其中包含恶意代码。
  2. 用户打开带有恶意代码的 URL 时，网站服务端将恶意代码从 URL 中取出，拼接在 HTML 中返回给浏览器。
  3. 用户浏览器接收到响应后解析执行，混在其中的恶意代码也被执行。
  4. 恶意代码窃取用户数据并发送到攻击者的网站，或者冒充用户的行为，调用目标网站接口执行攻击者指定的操作。 反射型 XSS 跟存储型 XSS 的区别是：存储型 XSS 的恶意代码存在数据库里，反射型 XSS 的恶意代码存在 URL 里。 反射型 XSS 漏洞常见于通过 URL 传递参数的功能，如网站搜索、跳转等。

DOM型

DOM型XSS的攻击步骤：

  1. 比如前端有有一个页面是通过获取input的value再显示到一个div中。
  2. 攻击者输入的一段script代码。
  3. 前端会将这一段script代码当成字符串解析并拼接到页面中，但是恶意代码将会执行。 DOM 型 XSS 跟前两种 XSS 的区别：DOM 型 XSS 攻击中，取出和执行恶意代码由浏览器端完成，属于前端 JavaScript 自身的安全漏洞，而其他两种 XSS 都属于服务端的安全漏洞。

XSS攻击的预防

通过以上可知，XSS有两大要素：

  1. 攻击者提交恶意代码
  2. 浏览器执行恶意代码

输入过滤

针对第一点，将用户的输入进行过滤，可不可行呢？ 答：攻击者一旦绕过前端，直接构造请求，就可以提交恶意代码了。 那么在服务端写入数据库之前进行过滤呢？ 答：服务端将转义后的数据发给前端，前端有时可以接受转义后的数据（比如HTML片段），有时是不能接受的(比如通过AJAX返回的Json数据赋给了一个变量)。 当然，对于明确的输入类型，例如数字、URL、电话号码、邮件地址等等内容，进行输入过滤还是必要的。

既然输入过滤并非完全可靠，我们就要通过“防止浏览器执行恶意代码”来防范 XSS。这部分分为两类：

  • 防止 HTML 中出现注入。
  • 防止 JavaScript 执行时，执行恶意代码。

反射型和存储型

存储型和反射型 XSS 都是在服务端取出恶意代码后，插入到响应 HTML 里的，攻击者刻意编写的“数据”被内嵌到“代码”中，被浏览器所执行。预防这两种漏洞，有两种常见做法：

  • 改成纯前端渲染，把代码和数据分隔开。
  • 对 HTML 做充分转义。

纯前端渲染

过程：

  1. 前端加载一个完全不依赖于业务数据的HTML.
  2. 之后加载JS代码，严格控制修改到前端的数据，比如明确分类：修改CSS类，修改属性类，修改文本类等。

HTML转义

如果拼接 HTML 是必要的，就需要采用合适的转义库，对 HTML 模板各处插入点进行充分的转义。

DOM型

在使用 .innerHTML、.outerHTML、document.write() 时要特别小心，不要把不可信的数据作为 HTML 插到页面上，而应尽量使用 .textContent、.setAttribute() 等。 DOM 中的内联事件监听器，如 location、onclick、onerror、onload、onmouseover 等，<a> 标签的 href 属性，JavaScript 的 eval()、setTimeout()、setInterval() 等，都能把字符串作为代码运行。如果不可信的数据拼接到字符串中传递给这些 API，很容易产生安全隐患，请务必避免。

其他预防措施

Content Security Policy

  • 禁止加载外域代码，防止复杂的攻击逻辑。
  • 禁止外域提交，网站被攻击后，用户的数据不会泄露到外域。
  • 禁止内联脚本执行（规则较严格，目前发现 GitHub 使用）。
  • 禁止未授权的脚本执行（新特性，Google Map 移动版在使用）。
  • 合理使用上报可以及时发现 XSS，利于尽快修复问题。

输入内容长度控制

对于不受信任的输入，都应该限定一个合理的长度。虽然无法完全防止 XSS 发生，但可以增加 XSS 攻击的难度。

其他

  • HTTP-only Cookie: 禁止 JavaScript 读取某些敏感 Cookie，攻击者完成 XSS 注入后也无法窃取此 Cookie。
  • 验证码：防止脚本冒充用户提交危险操作。

相关重点总结

输入过滤，转义，纯前端渲染，验证码，HttpOnly

CSRF

CSRF简介

利用受害者在被攻击网站已经获取的注册凭证，绕过后台的用户验证，达到冒充用户对被攻击的网站执行某项操作的目的。

一个典型的CSRF攻击有着如下的流程：

  • 受害者登录a.com，并保留了登录凭证（Cookie）。
  • 攻击者引诱受害者访问了b.com。
  • b.com 向 a.com 发送了一个请求：a.com/act=xx。浏览器会…
  • a.com接收到请求后，对请求进行验证，并确认是受害者的凭证，误以为是受害者自己发送的请求。
  • a.com以受害者的名义执行了act=xx。
  • 攻击完成，攻击者在受害者不知情的情况下，冒充受害者，让a.com执行了自己定义的操作。

CSRF防护

同源检测

CSRF大多来自第三方网站，那么禁止外域的请求就可以了。如何判断外域？ 在HTTP协议中，每一个一步请求都会携带两个Header，如下

  • Origin Header
  • Referer Header 简单来说就是通过验证上面两个参数来判断是否同源。

CSRF Token

  • 首先，用户打开页面的时候，服务器需要给这个用户生成一个Token，该Token通过加密算法对数据进行加密,显然在提交时Token不能再放在Cookie中了为了安全起见Token最好还是存在服务器的Session中，之后在每次页面加载时，使用JS遍历整个DOM树，对于DOM中所有的a和form标签后加入Token。这样可以解决大部分的请求，但是对于在页面加载之后动态生成的HTML代码，这种方法就没有作用，还需要程序员在编码时手动添加Token。
  • 页面提交的请求携带这个Token
  • 服务端验证Token

Samesite Cookie属性

从源头上解决这个问题，Google起草了一份草案来改进HTTP协议，那就是为Set-Cookie响应头新增Samesite属性，它用来标明这个 Cookie是个“同站 Cookie”，同站Cookie只能作为第一方Cookie，不能作为第三方Cookie，Samesite 有两个属性值，分别是 Strict 和 Lax： Samesite=Strict 种称为严格模式，表明这个 Cookie 在任何情况下都不可能作为第三方 Cookie Samesite=Lax 这种称为宽松模式，比 Strict 放宽了点限制：假如这个请求是这种请求（改变了当前页面或者打开了新页面）且同时是个GET请求，则这个Cookie可以作为第三方Cookie Samesite的问题：子域不能使用

ESLint的作用

  1. 配置注释： 使用JavaScript注释将配置信息直接嵌入到文件里
  2. 配置文件： 有多种文件类型，以.eslintrc.js为常见，还有.eslintrc.yaml,.eslintrc.json等
  3. 在package.json文件中创建eslintConfig属性

.yaml：YAML任然是一种标记语言，多用于配置文件

  1. 选择器选项（parserOptions）

  • Plugin 从前到后执行
  • Preset 从后到前执行

preset 的逆向顺序主要是为了保证向后兼容，因为大多数用户的编写顺序是 ['es2015', 'stage-0']。这样必须先执行 stage-0 才能确保 babel 不报错。因此我们编排 preset 的时候，也要注意顺序，其实只要按照规范的时间顺序列出即可。

babel深入

babel的工作流程：

Babel运行共三个阶段： 解析， 转化，生成 Babel插件也相应分为语法插件和转译插件，前者再解析时工作，后者在转化时工作，大部分插件也就是我们要关注的插件是转译插件

解析：

将代码解析成抽象语法树（AST），每个js引擎（比如Chrome浏览器中的V8引擎）都有自己的AST解析器，而Babel是通过Babylon实现的。在解析过程中有两个阶段：词法分析和语法分析，词法分析阶段把字符串形式的代码转换为令牌（tokens）流，令牌类似于AST中节点；而语法分析阶段则会把一个令牌流转换成 AST的形式，同时这个阶段会把令牌中的信息转换成AST的表述结构。

转化：

在这个阶段，Babel接受得到AST并通过babel-traverse对其进行深度优先遍历，在此过程中对节点进行添加、更新及移除操作。这部分也是Babel插件介入工作的部分。

生成：

将经过转换的AST通过babel-generator再转换成js代码，过程就是深度优先遍历整个AST，然后构建可以表示转换后代码的字符串。
